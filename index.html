03 / 12

THREE AGENTS

三个 Agent，各司其职

🗺️
Planner

规划者

探索应用，产出 Markdown 格式的测试计划

输入

任务描述 + seed 文件

输出

specs/*.md 测试计划

⚡
Generator

生成者

把测试计划转成可执行的 Playwright 测试代码

输入

specs/ 下的 Markdown 计划

输出

tests/*.spec.ts 测试代码

🔧
Healer

修复者

跑失败的测试，自动找原因，自动打补丁修好

输入

失败的测试文件

输出

修复后的测试代码

04 / 12
Planner · 规划者

先探路，
再规划

Planner 会先跑 seed 测试配置环境，然后在此基础上探索应用，把 seed 文件当作后续生成测试的风格模板。

1

接收清晰的任务描述

如"给访客结账流程生成测试计划"

2

探索应用，了解页面结构

自动识别可测试的元素和流程

3

输出人类可读的测试计划

Markdown 格式，场景、步骤、预期一目了然

// seed.spec.ts 示例

import { test, expect } from './fixtures';


test('seed', async ({ page }) => {

// 在这里处理登录、初始化等准备工作

// 这个测试本身不需要有断言

});

💡

Seed 文件是整个流程的基础

应用需要登录但 seed 没处理，Planner 探索时会撞墙

05 / 12
Generator · 生成者

计划 →
可执行代码

Generator 在生成过程中会实时验证——真的操作浏览器，确认选择器有效、断言合理。生成完就是能跑的代码（大部分情况）。

📥

读取 specs/ 下的 Markdown 计划

🔍

实时操作浏览器验证选择器

📤

输出 tests/*.spec.ts，与计划一一对应

// 告诉 Generator：

"把 specs/basic-operations.md

里的测试计划生成为测试代码"


// Generator 输出：

tests/

├── add-valid-todo.spec.ts

├── edit-todo.spec.ts

└── delete-todo.spec.ts

关键特性

生成过程中真的操作浏览器，确认选择器有效、断言合理

06 / 12
Healer · 修复者

测试挂了？
自动修

这是三个 Agent 里最省心的一个。跑失败的测试交给他，自动回放、诊断、打补丁。

1

重新回放失败的测试步骤

2

查看当前 UI，找到对应的元素或流程

3

提出修复方案（更新选择器、调整等待、修改数据）

4

重新跑测试，直到通过

// 告诉 Healer：

"帮我修复 tests/add-valid-todo.spec.ts

里失败的测试"

⚠️

Healer 的能力边界

选择器大范围改动、功能逻辑变了、接口 schema 变了 —— 这些靠 Healer 修不了。它处理的是"能跑但跑失败了"这类问题。

多次修复仍不行时，Healer 会标记为 skip 并注明功能本身可能有问题

07 / 12

WORKFLOW

完整工作流：Agentic Loop

三个 Agent 接力，你主要在审核，不在执行

1

写好 seed.spec.ts

处理环境初始化（登录、数据准备等）

2

Planner 探索应用，输出测试计划

"给购物车模块生成测试计划" → specs/cart-operations.md

3

人工审核计划（可选）

检查 specs/ 下的计划文件，觉得 OK 就放行

4

Generator 生成可执行测试代码

计划 → tests/*.spec.ts，实时验证选择器有效

5

跑测试，有失败的？Healer 去修

自动回放、诊断、打补丁，修不好标记 skip

6

全部通过，合并进代码仓库 ✅

08 / 12

QUICK START

快速开始

一条命令初始化，之后在 AI 编码工具里直接用

初始化 Agent 定义文件

# VS Code 模式（默认）

npx playwright init-agents


# Claude Code 模式

npx playwright init-agents --loop=claude

生成的目录结构

repo/

.github/ # Agent 定义文件

specs/ # 人类可读的测试计划

basic-operations.md

cart-operations.md

tests/ # 生成的测试代码

add-valid-todo.spec.ts

seed.spec.ts # 种子测试

playwright.config.ts

💡

VS Code 版本要求

需要 v1.105（2025 年 10 月 9 日之后）才能完整支持 Test Agents

⚠️

版本升级注意

Playwright 版本更新后要重新跑 init-agents，定义文件不会自动同步

计划与测试一一对应

测试失败了，去 specs/ 找对应计划，看是需求层面还是代码层面的问题。出了问题方便追溯。

三个 Agent 可灵活组合

可以单独用、按顺序接力、或串成自动化循环。不需要全流程走完，按需调用即可。

BEST PRACTICES

实战注意事项

🔥

Seed 文件一定要写好

整个 Agent 流程的基础。应用需要登录但 seed 没处理，Planner 探索时会撞墙。花时间把 seed 写扎实，后面事半功倍。

🔄

定义文件跟着版本走

每次升级 Playwright 后重新跑 init-agents，定义文件不会自动同步。

🛡️

Healer 不是万能的

选择器大范围改动、功能逻辑变了、接口 schema 变了 —— 这些需要人工介入。Healer 处理的是"能跑、跑失败了"的情况。

👀

生成的代码要过一遍眼

断言力度够不够、边界情况有没有覆盖、测试间是否有依赖 —— 这些仍需人判断。AI 生成 ≠ 免审查。
