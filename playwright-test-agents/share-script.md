# Playwright Test Agents 分享稿

> 配套 PPT：`playwright-test-agents-ppt.html`
> 预计时长：20 分钟
> 定位：技术分享，面向有一定前端/测试经验的开发者

---

## P1 · 封面（30s）

大家好，今天分享一个我最近发现特别有意思的东西——Playwright 官方推出的 Test Agents。

一句话概括就是：**三个 AI Agent 帮你全自动写测试**。你只需要告诉它测什么，它自己探索应用、自己写代码、跑失败了还能自己修。

接下来大概 20 分钟，我会把这三个 Agent 是什么、怎么用、什么时候该用，讲清楚。

---

## P2 · 痛点引子（2min）

在正式讲之前，我们先对齐一下痛点。写测试这件事，大家心里都知道重要，但现实中往往是这四种状态：

第一种——**手动写测试太耗时**。一个中等规模的项目，随随便便几百个测试用例，逐个手写成本巨大。 deadline 一紧，"赶完功能就算了"是常态。

第二种——**页面一改测试就挂**。选择器变了、文案改了、流程调了……修测试比写测试还痛苦。而且往往是别人改的页面，你不知道，CI 一跑红了才发现在查半天。

第三种——**测试覆盖率上不去**。核心流程勉强覆盖住，但边界情况、异常路径、各种组合场景，永远没人管。覆盖率报告看着就头疼。

然后第四种是我们今天的重点——**AI 能不能帮我干这些活？** Playwright 官方的回答是：可以，而且全程不用你盯着。

---

## P3 · 什么是 Test Agents（1.5min）

好，那 Test Agents 到底是什么？

它是 Playwright 官方推出的 AI 测试系统，包含三个专门干测试的 AI Agent。这三个 Agent 不是一个东西的三种模式，而是**三个独立的角色**，各有各的职责，形成一个完整的测试自动化链路。

Planner 是规划者，负责探索你的应用然后输出测试计划。Generator 是生成者，把计划变成可执行的代码。Healer 是修复者，测试挂了自动修。

它们的关系是这样的：Planner 输出给 Generator，Generator 输出给 Healer。你可以单独用其中一个，也可以串成完整流程。

官方有句话说得好——以前是你写代码让机器自动跑，现在是你说需求让 AI 自动生成代码再自动跑。这是对"测试自动化"的重新定义。

---

## P4 · 三个 Agent 总览（2min）

我们逐个看一下。这一页是总览，后面会分别展开。

**Planner，规划者。** 你给它一个清晰的任务描述，比如"给访客结账流程生成测试计划"，再加一个 seed 文件处理登录初始化，它就会自己探索你的应用，然后输出一个人类可读的 Markdown 格式的测试计划。每个测试场景、步骤、预期结果都写得清清楚楚。

**Generator，生成者。** 它接收 specs 目录下的 Markdown 计划文件，然后转成可执行的 Playwright 测试代码。关键是它生成过程中会真的操作浏览器去验证，确认选择器有效、断言合理。所以生成完的代码大部分情况下是能直接跑的。

**Healer，修复者。** 这是三个里面最省心的。你把跑失败的测试丢给它，它会重新回放、看 UI、找原因、打补丁。如果多次修复还过不了，它会标记为 skip 并说明原因——可能是功能本身有问题，不是测试的问题。

---

## P5 · Planner 详解（2min）

展开讲一下 Planner。

Planner 的工作方式是：先跑你提供的 seed 测试把环境配好，比如登录、初始化数据这些。然后在此基础上，它开始探索你的应用。探索完了，输出一份 Markdown 格式的测试计划到 specs 目录。

右边这个是 seed 文件的示例。非常简单，就是一个普通的 Playwright 测试，但不需要有断言。它的作用就是帮 Planner "进门"——把环境准备好。

这里有个关键点：**seed 文件是整个流程的基础**。如果你的应用需要登录，但 seed 里没处理，Planner 探索的时候就会撞墙。所以花时间把 seed 写扎实，后面事半功倍。而且 Planner 会把 seed 文件当作后续生成测试的风格模板，所以你在 seed 里写的代码风格，会影响后面所有生成的测试代码。

---

## P6 · Generator 详解（2min）

接下来是 Generator。

Generator 的输入是 specs 目录下的 Markdown 计划文件，输出是 tests 目录下的 .spec.ts 文件，跟计划里的场景一一对应。

它的最大特点是**实时验证**。它不是凭空生成代码，而是真的会操作浏览器，去确认选择器能不能找到元素、断言合不合理。这就意味着生成出来的代码大部分情况是能跑的，不是那种 AI 生成完还得你大改的半成品。

使用方式也很简单，你直接告诉它"把 specs/basic-operations.md 里的测试计划生成为测试代码"就行。它会输出对应的测试文件，比如 add-valid-todo.spec.ts、edit-todo.spec.ts 这些。

---

## P7 · Healer 详解（2min）

然后是 Healer，也是我觉得最实用的一个。

使用方式就是一句话："帮我修复 tests/add-valid-todo.spec.ts 里失败的测试。"

它的工作流程是四步：先重新回放失败的测试步骤，然后查看当前 UI 找到对应的元素或流程，接着提出修复方案——可能是更新选择器、调整等待时间、修改测试数据，最后重新跑测试直到通过。

但要注意 Healer 的**能力边界**。它处理的是"能跑、跑失败了"这类问题。如果你的选择器大范围改了、功能逻辑本身变了、或者接口的 schema 变了，这些 Healer 修不了，需要人工介入。

另外如果多次修复仍然过不了，Healer 会标记为 skip 并注明"功能本身可能有问题"。这个设计挺合理的，等于是在帮你做了一层故障定位。

---

## P8 · 完整工作流（2min）

把三个 Agent 串起来看完整的工作流。

第一步，写好 seed.spec.ts，处理环境初始化。

第二步，告诉 Planner 你要测什么，比如"给购物车模块生成测试计划"，它会输出 specs/cart-operations.md。

第三步是可选的——人工审核一下计划文件。觉得 OK 就放行，觉得有问题可以调整。

第四步，告诉 Generator 把计划生成为测试代码。它会实时验证然后输出可执行的 .spec.ts 文件。

第五步，跑测试。有失败的？交给 Healer 去修。

第六步，全部通过，合并进代码仓库。

你可以看到，整个流程里人的角色主要在**审核**，不在**执行**。这就是官方说的 Agentic Loop——三个 Agent 接力，你在关键节点把关就行。

---

## P9 · 快速开始（2min）

那具体怎么上手呢？就一条命令。

`npx playwright init-agents`，默认是 VS Code 模式。如果你用 Claude Code，加 `--loop=claude` 参数。

这条命令会在项目的 .github 目录下生成 Agent 定义文件，告诉 AI 有哪些工具可以用、怎么配合工作。

右边是生成之后的目录结构。.github 是 Agent 定义文件，specs 是人类可读的测试计划，tests 是生成的测试代码，seed.spec.ts 是种子测试。

有几个注意事项。第一，VS Code 需要 v1.105 以上版本才能完整支持，2025 年 10 月之后的版本就行。第二，Playwright 版本更新后要重新跑 init-agents，定义文件不会自动同步。第三，计划和测试是一一对应的，出了问题方便追溯——测试挂了去 specs 找对应计划，看是需求层面还是代码层面的问题。

---

## P10 · MCP vs CLI vs Agents 对比（2min）

有些人可能会问，Playwright 之前不是已经有 MCP 和 CLI 了吗？Test Agents 跟它们什么关系？

我们看这个对比表。

MCP 的本质是给 AI 装了"眼睛"，你指挥 AI 执行，交互方式是对话式的，全程你都在场。Token 消耗比较高因为要传页面快照。适合写单个测试或者边调试边写。

CLI 是给 AI 装了"手脚"，AI 自己下命令自己执行，适合接入自动化流水线。Token 消耗低，但不太适合互动调试。

Test Agents 是一个自主 AI 系统，你给目标它完成全程，交互方式是目标式的。Token 消耗中等因为有中间产物。适合做项目级的测试覆盖建设。人的介入只在关键节点审核就行。

一句话选型原则：**写单个测试找 MCP，搭自动化流程用 CLI，建整体测试覆盖上 Agents。**

---

## P11 · 注意事项（1.5min）

最后分享几条实战注意事项。

第一，**Seed 文件一定要写好**。整个 Agent 流程的地基就是 seed 文件。应用需要登录但 seed 没处理，Planner 探索时直接撞墙。花时间把 seed 写扎实，后面事半功倍。

第二，**Agent 定义文件跟着版本走**。Playwright 升级后要重新跑 init-agents，不然可能用不了新功能或者行为不一致。

第三，**Healer 不是万能的**。它处理的是"能跑但跑失败了"的情况。如果是功能逻辑变了、接口 schema 变了，该人工介入就人工介入。

第四，**生成的代码要过一遍眼**。AI 生成的测试在断言力度、边界情况、测试间依赖这些方面，仍然需要人的判断。AI 生成不等于免审查，把它当成一个效率极高的初稿，你来做终审。

---

## P12 · 总结 & Q&A（30s）

最后快速回顾一下核心要点。

Planner 负责探索应用、输出人类可读的测试计划。Generator 把计划变成可执行的 Playwright 测试代码。Healer 在测试失败时自动修复，修不好就标记原因。

三个 Agent 可以串成完整流程，也可以按需单独调用。

起步就一条命令：`npx playwright init-agents`。
